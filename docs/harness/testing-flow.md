# 测试流程总纲（testing flow）

> 测试全流程的**唯一真相源**。`hc-test` 总监 / 各测试子 agent / ADR-0014 都**引用本文、不复制**——改流程只动这里。
> 北极星愿景见 [testing-flow-vision](../superpowers/specs/2026-06-29-testing-flow-vision.md)（`status: vision`，原始口述）；本文是**可执行总纲**（`status: active`）。

## 定位

- `hc-test` = **测试总监**（编排逻辑，**主 agent 当总监**），按本总纲调度专职子 agent。
- **分阶段【实现】、整体【发布】**：e2e + api + 脚本全套实现完才发布；未发布前各段是 WIP，不单独 merge。
- **贯穿原则：每步解耦、用户可跳过任意一步**（可以没用例、没脚本……都行）。

## 总监怎么派活

- **默认（A）：按手上产物 + 到了哪一步自动选场景**——有 PRD 先能做 e2e 用例；有研发方案 + 接口契约才做 api 用例；开发彻底结束才写脚本。
- **用户指令最高优先级**：随时点名做哪段 / 跳过哪段（覆盖默认）。沿用 `hc-prd` 总监"默认编排 + 用户覆盖"模式。

## 全流程（目标态，按时序 + 依赖）

1. **PRD 产出后** → 可做 **e2e 用例**；**并行**产出**研发方案**（含接口契约）。
2. **研发方案产出后**（有接口契约）→ 可做 **api 用例**。
3. 用例产出 → **review 校验**。
4. **开发完成** + 有 routelist → **接口契约对照**。
5. **开发彻底结束** → **测试脚本**（api + e2e）。
6. 脚本 → **统一回归**（先本需求、再统一，不一上来跑带过往的全量）→ 不通过 → 回改到通过。

## 场景 × 实现状态

| 场景 | 触发 | worker → reviewer | 状态 |
|---|---|---|---|
| e2e 用例 | 有 PRD | `hc-e2e-qa` → `hc-e2e-reviewer` | ✅ 本期实现 |
| api 用例 | 有研发方案 + 接口契约 | `hc-api-qa` → `hc-api-reviewer` | 🔒 占位·待研发方案 |
| 接口契约对照 | 开发完成 + routelist | 总监调度 | 🔒 占位 |
| 测试脚本 | 开发彻底结束 | `hc-script-impl` → `hc-script-reviewer` | 🔒 占位·待 sandbox |
| 统一回归 | 脚本就绪 | 总监调度 | 🔒 占位 |

> 占位 = 全量设计已留位，加时**填空、不重构** skill 形态。

## e2e 用例线（本期实现）

### 输入优先级（缺则略过）

| 优先级 | 产物 | 拿它干嘛 |
|---|---|---|
| ① 最高 | **AC 验收点**（`user-stories.md`） | 可观测的通过/失败判据 → 用例的"预期" |
| ② | **功能点清单 FP**（`feature-points.md`） | 枚举功能/交互点 → 覆盖广度（每个交互点都有用例） |
| ③ | **用户故事 US** | 业务闭环 / 场景上下文 |
| ④ | **PRD 正文**（`prd.md`） | 交互细节、四态、成败的具体表现 |

> AC/FP 定"测什么、覆盖够不够"（硬覆盖）；US/PRD 定"怎么走、长什么样"（场景细节）。哪个产物没有就略过、用现有的，不卡。

### `hc-e2e-qa`（写用例）

- 走**完整业务闭环**（开页 → 点按钮 → 填表 → 提交 → 看状态），不只主流程；playwright / 谷歌插件级模拟点击、UI 交互。
- **每个 UX 交互点 × {成功, 失败, 边界} 都要有用例**；**等价类 / 边界值不可略**。
- **成功**用例写"成功该怎样"（跳转 / 状态 / 数据）；**失败**用例写"失败该怎样"——必须**受控**（不崩、不白屏、不 5xx 级）。
- 预期**锚唯一真实信号**（rule-0009），不靠脆弱的 toast 文案。
- 每条用例用 `covers:` 声明覆盖哪些 `AC-NN`/`FP-NN`（覆盖唯一真相源，rule-0014）。
- 套模板 `templates/e2e-test-case.md`；产物落 `docs/test-cases/<id>/`；**只写用例、不跑**（rule-0014）。
- 以上要求**写进 `hc-e2e-qa` 子 agent 的上下文**（system prompt），不只靠模板——模板只定产物形状，约束主体在上下文。

### `hc-e2e-reviewer`（审用例）—— 4 块

1. **主观覆盖率审查**：判断覆盖**真不真够**（漏没漏没进矩阵的交互点、是不是只测浅层 happy path、关键场景组合缺没缺）——用判断补机检盲区。
2. **引用源符合 + 理解偏差**：每条用例**忠不忠于它 `covers:` 的源**（AC/FP/PRD）；回到源头对照，看 `hc-e2e-qa` 有没有**读错需求、测歪**（语义偏差，如 AC 说"5 次锁定"用例测成"3 次"——covers 挂着、机检过，但语义跑偏）。
3. **用例质量**：等价类划分、边界值取真边界（下界 / 下界-1 / 上界 / 上界+1）、预期锚定（rule-0009）、牵强 / 空转用例。
4. 结构层不重复（交给机检，见下）。
- **只评不改**；出结构化清单 → 总监派 `hc-e2e-qa` 回改 → 复审到零。判据 = 考题 015 + rule-0014 + rule-0009。要求同样写进 reviewer 上下文。

### 用例模板（`templates/e2e-test-case.md`）

顶部 **"交互点 × 类型 覆盖矩阵"**（一眼看漏没漏）；每条用例字段：`covers:` / 交互点 / 类型（正常·边界·异常 + 等价类标注）/ 数据（标等价类·边界值）/ 前置 / 步骤（点击流程）/ 预期（成功 or 失败，分开写死）。

### 两层覆盖防线

| 谁 | 查什么 | 判据 |
|---|---|---|
| **机器**（`test-cases-audit`） | **结构**：覆盖矩阵格子填没填 / 标没标、`covers:` 闭合无悬空 | rule-0014 |
| **`hc-e2e-reviewer`** | **判断**：覆盖够不够、懂没懂源、忠不忠、质量牵不牵强 | 考题 015 + rule-0009 |

矩阵硬机检带 **"无·理由"逃生口**：每个交互点的 成功 / 失败 格，要么有 `TC-NN`、要么显式标 `无·理由:<…>`；**空着没理由 = 红、标了理由 = 过**（防误伤；机器只查结构、不碰"用例好不好"——那归 reviewer；要跳过也是 QA 写了理由、可追溯）。

## api 用例线（占位·api worker 待建）

- **接口契约来源**：由 `hc-design` 产出（`docs/designs/<id>/api-contract.md`，ADR-0015）——这就是本线的「研发方案 + 接口契约」前置，现已可产出；api 用例 worker（`hc-api-qa` / `hc-api-reviewer`）随后建。
- 对齐接口契约：正常请求的**参数 / 响应 / 字段类型 / 边界值 / 等价类**；异常**覆盖契约里列的约定内错误**——业务码 / 校验 400·422 / 鉴权 401·403 / 约定服务态（如 503 DB_UNAVAILABLE，契约列了就要覆盖）；**未约定的未预期故障**（裸 500 / panic）不在契约、不为它编用例。**按"约定 / 未约定"分，不按状态码段一刀切**。
- 接口契约本身含正常 + 异常约定，用例与契约对齐。

## 接口契约对照（占位·开发完成 + routelist）

- routelist 新增接口 vs 接口契约是否一致；不一致 → 复核根因：**契约不足** → 补契约 + 补用例；**实现不足** → 补实现。

## 测试脚本（占位·待 sandbox）

- api + e2e 脚本；**每需求独立**（不和过往混）；与用例**一一对照**；**单独审核**（`hc-script-reviewer`）。

## 回归（占位）

- 先本需求、再统一回归；**不一上来跑带过往的全量**。不通过 → 回改到通过。

## 引用约定

- 各 skill / 子 agent / ADR **引用本文对应小节，不复制**流程叙述。改流程只改本文（唯一真相源）。

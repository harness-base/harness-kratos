# eval 015 — 测试用例覆盖完整性与质量（rule-0014 / ADR-0008）

## 评什么

`test-case` 产出的测试用例够不够"覆盖了需求、用例是真的"——重点：**AC + 功能点 FP 双轴全覆盖**、**用例真覆盖语义（非空壳凑数）**、**正常 / 边界 / 异常齐**、**`covers:` 是唯一真相源**、**不碰执行结果**。覆盖有洞、用例是空壳、只测正常路径、`covers:` 与用例对不上、把"过没过"混进来，都打回。

## 检查清单（逐条查）

- **AC/FP 双轴全覆盖（硬，机检兜底）**：每条 `AC-n`、每个 `FP-n` 都被 ≥1 条用例 `covers:` 覆盖？`covers:` 无悬空引用（指向未声明的 AC/FP）？此项 `scripts/test-cases-audit.sh` 已机器校验，这里抽验确认没绕过（如把 AC 漏写出清单来假装覆盖全）。**抽验段标题契约**：AC/FP 声明确实落在含「验收点」「功能点」的 `## ` 段内、`covers:` 在 `## 用例`/`## 测试用例` 段内（套了模板就没问题）；若机检报「未覆盖 / 悬空」，先排除是**段标题被改名 / 声明落段外 / `covers:` 折行**导致的静默漏算或误报，而非真缺覆盖。
- **用例真覆盖语义（软）**：每条用例的步骤 + 预期是否**真验到**它 `covers:` 的那条 AC/FP，而非挂个 id 凑数的空壳？预期是否**可观察可判定**（不是"功能正常""支持 X"这类不可验证措辞）？空壳 / 不可观测 = fail。
- **正常 / 边界 / 异常齐**：每条 AC/FP 是否有正常用例 + **必要的边界 / 异常**用例？只覆盖 happy path = fail（除非该点确无边界 / 异常，需说明）。
- **`covers:` 唯一真相源**：覆盖关系是否只靠各用例 `covers:` 维护、没有另立一份手维护映射表与之并存（两份会漂，rule-0012）？模板里的覆盖映射表若存在，须标注"权威以 covers 为准"。
- **不碰执行结果**：产物是否只管"用例齐不齐 / 覆盖全不全"，**没有**掺入"过没过 / 通过率 / 跑出来的代码覆盖率"（那是另一段机制）？混入即 fail。
- **来源可追**：AC/FP 是否誊录自真实需求来源（用户口述 / `user-stories.md` / feature 包），而非凭印象臆造？臆造的验收点 = fail。
- **登记一致**：用例集登记进 `docs/test-cases/index.yaml`（条目含 `dir: <id>`）、目录与账本一致（机器侧 `test-cases-audit` 已查，这里抽验）？

## 输出

verdict（pass / fail）+ 命中问题的 `文件:位置` 与具体证据（哪条 AC/FP 没被覆盖、哪条用例是空壳、哪条预期不可观测、哪点缺边界 / 异常、是否混入了执行结果）+ 修法建议。

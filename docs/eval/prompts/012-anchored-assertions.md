# eval 012 — 验收断言/声称的保证是否由唯一真实证据守护（rule-0009）

## 评什么

候选交付里"绿了/声称成立"的东西，会不会其实没证明真行为。四种翻车：① 断言 grep 裸串命中**请求访问日志回显的入参**，"发了"当"处理了"（S3 实案：消息 100% 丢失但 14 项全 PASS，见 `task-reviews/20260612T041709Z-kratos-base-s3/`）；② 共因污染（一个信号多条因果）；③ 超时竞态（断言随超时配置翻盘）；④ 声称的保证无守护测试（注释撒谎）。②③④ 见下方专项清单（S5/S6 实战补充）。

## 检查清单

- **锚定性**：断言是否锁定**只能由处理方产出**的结构化字段 + 业务 id（如 `"consumer":"received"` 且 `key==<事件 id>`）？裸串全文 grep 一律打回。
- **无兜底**：有没有"精确匹配不到就接受任意相关行/只要有 consumer 行就算过"之类的降级分支？有即 fail。
- **回显排查**：被断言的内容（payload/id）会不会同时出现在请求访问日志、回声响应里？会，则断言必须能区分两者。
- **对端证据**：跨系统链路（MQ/队列/异步任务）是否有对端侧证据（队列深度、消费者数、外部系统台账）佐证"到达/处理"，而非只看本进程日志？
- **抽查实证**：抽 1 条关键断言，在日志/产物里找出它实际命中的行——确认那行真是处理方写的。

### 三类假阳性形态（rule-0009 S5/S6 补充，逐条查）

- **A. 共因污染**：被断言的可观测量（如 `/readyz` 503）是否**只有目标行为能触发**？若同一步骤既"注入变更"又"停了某容器/进程"，503/失败可能来自旁因——断言无法区分目标行为，fail。正解是制造唯一因果（如改依赖地址但不停其容器）。
- **B. 超时竞态/配置敏感**：这条断言会不会因为"改了某超时/时序配置"就翻盘？若结论依赖"阻塞耗光某 ctx→超时"这类竞态（典型失败信息 `context deadline exceeded` 而非干净业务/连接错误），它锚的不是真行为，fail。正解是锚定与时序无关的产出方痕迹（如日志 BEFORE/AFTER 计数对比）。
- **C. 声称的保证无测试守护**：代码注释/文档里"绝不超过 X / 总是 / never / always"这类**行为保证**，有没有一条违反即失败的测试钉死？没有就是"注释会撒谎"，要求补守护测试、或如实降级措辞（如"快速失败"→"有界失败 ≤request_timeout"）。

## 输出

verdict（pass / fail）+ 命中问题的 `脚本:行` / `注释:行` 与实际被匹配内容/失效场景 + 修法建议。

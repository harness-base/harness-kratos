# ADR-0007：产出需求流程重做——分阶段确认门 + 真相/需求源优先级 + 功能点覆盖

## 背景

`prd-elicitation`（产出需求，ADR-0003 确立为 feature-delivery 上游）现有流程是"多轮对话 → 直接产 PRD + 原型"：一步跨度大、没有中间确认门，对"现状/假设"的处理只在步骤里零散提了句"不静默假设"。且 `docs/prds/` 至今是空账本——流程设计完整但**零实战**。

讨论确认：产出需求应是一条**有约束、分阶段、每阶段用户确认**的工作流，并把"不靠假设、缺信息就去查事实源"从一个可选前置步骤升级为**贯穿全程的约束**。

## 决策

1. **"现状勘察"从步骤 → 贯穿约束**：任何信息不足，去查仓内代码/资产/文档（事实源）或摆出来问用户确认，**不拿"已有信息 + 想要的结果"硬假设**；与原"不静默假设"合并为一条贯穿规则。前置只保留"用户指定了信息源就先读"，不单列为正式步骤。

2. **两套优先级（写进 skill）**：
   - **真相源**（什么是真的）：① 用户当场直说 > ② 仓内现有代码/资产/现状 > ③ 用户提交的信息源（目录/文档/图）。
   - **需求源**（要做成什么）：① 用户当场直说 > ② 用户提交的信息源 > ③ 现状。
   - 依据：论事实，代码现状优先于提交文档（代码是地面真相、文档可能过期）；论需求，提交源优先于现状（现状是起点不是目标）。

3. **5 步分阶段工作流，每步带"多轮沟通 + 用户确认门"**：
   ① 需求澄清（引导式多轮对话，沿用现有 skill）→ ② 产出用户故事——**一等持久产物** `docs/prds/<id>/user-stories.md`（套 `templates/user-story.md`，与 `prd.md` 分开存），**先于 PRD 产出**并经用户审核确认（approved 才进 PRD）；它是后续 PRD / 原型 / 评审**统一对齐"需求"的事实视角**（覆盖追溯以它为锚）→ ③ 产出 PRD（须符合已确认用户故事，套 `templates/prd.md`，带功能点清单，用户确认）→ ④（可选）产出原型（带 UI 属性、与现有前端内容/风格一致、不复刻还原（见 ADR-0003）；无界面/生成不了则跳过、能则先问用户；用户确认）→ ⑤ 对抗式完整性评审（loop engineering，审→修→再审到零）。

4. **功能点覆盖（软要求）**：三级双向可追溯——用户故事 ⟷ 功能点清单 ⟷ PRD 内容；标准为 100% 覆盖（每个用户故事 ≥1 功能点、每个功能点可追到用户故事且在 PRD 有描述、PRD 无孤儿内容）。**软**：靠 PRD 内一张映射表 + 对抗评审自查 + 写作指南，**不做机器闸**（`prds-audit` 暂不卡覆盖率，留待以后硬化）。

5. **新增/调整产物**：新建 `templates/user-story.md`；新增 `prd-elicitation/references/prd-writing.md`（PRD 写作指南：模板为底、与用户故事/功能点呼应、人类可读、业务逻辑清晰、旧场景简单一句话 / 复杂附引用来源让用户自查）；`templates/prd.md` 增"功能点清单 + 映射"段。

## 受影响的 skill（rule-0007）
- skill：prd-elicitation ／ 是否已更新：**是**——本 ADR 即其流程重做（5 步 + 贯穿约束 + 两套优先级 + 覆盖），随实现重写 `SKILL.md` 并 bump version/last_reviewed。
- skill：feature-delivery ／ 是否已更新：**否**——下游不变，仍与产出需求松耦合（PRD 非强制），衔接口（PRD 派生 feature 包）未改。
- skill：其余（context-loading / add-rule / doc-sync / git-workflow / bugfix / self-evolution）／ 否——与产出需求流程无关。

## 备选方案
- **保持现状（一步到 PRD + 原型）**：拒。跨度大、无中间确认门、无用户故事阶段、无覆盖保证；零实战已暴露它"纸面完整"。
- **覆盖率做成硬机器闸（`prds-audit` 卡覆盖率%）**：暂拒。先软，避免流程没跑顺就过早僵化；复发 / 需要时再硬化（soft→hard）。

## 影响
- PRD 产出更可控、可追溯、可读；多了"用户故事"确认阶段（轮次变多，但每步更稳、返工更早）。
- 原型可选化，适配无界面项目。
- 护栏需对齐：rule-0010（PRD 标准）、eval 题 013（完整性）、`prds-audit`（必备章节可能加"功能点清单"）随实现调整。
- 仍与实现体系松耦合（ADR-0003 的上游定位不变）；本 ADR 是对其**流程**的细化，不推翻 0003。

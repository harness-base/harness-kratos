# 测试闭环 — 最终目标（愿景）

> 用户 2026-06-29 口述的完整测试流程愿景。这是**目标态**，不是当前实现——分阶段落地。
> **贯穿原则：每一步都不耦合、用户可跳过任意一步**（可以没测试用例、没测试脚本……都行）。

## 完整流程（目标态，按时序 + 依赖）

1. **PRD 完整产出后** → 可做 **e2e 测试用例**；**同时并行**产出**研发方案**（含接口契约）。
2. **研发方案产出后**（有了接口契约）→ 可做 **api 测试用例**。
3. 用例产出后 → **review 校验**（见下）。
4. **开发完成后** → 若有 routelist，**接口契约对照**（见下）。
5. **开发彻底结束后** → 写**测试脚本**（api + e2e）。
6. 脚本写完 → **统一回归**；不通过 → 回改到通过。

## 两类测试用例（分开写、分开测）

**api 测试用例**（对齐接口契约）：
- 正常请求：参数、响应、字段类型、**边界值、等价类**。
- 异常：业务 / 验证异常（**422、400** 等）要覆盖；**500 一类不覆盖**。
- 接口契约本身也得包含这些（正常 + 异常的约定）——用例与契约对齐。

**e2e 测试用例**（完整业务闭环）：
- 不只主流程——**完整业务闭环**。
- 含 **playwright / 谷歌插件**的模拟点击、UI 交互。
- 每个小功能、每个按钮、等价类、边界值都要覆盖。

## review（校验用例）
- 一个 **review subagent**，对两类用例校验：
  - 能**脚本校验**的加上：功能点清单 / 用户故事 / 接口契约 三方对照。
  - **agent 主观判断**的覆盖率。
- 用例要完整，**不能敷衍**。

## 开发完成后：接口契约对照（若有 routelist）
- 对照 routelist：开发新增的接口 vs 接口契约是否一致。
- 不一致 → 复核根因：**前期契约不足** 还是 **实现有问题**？
  - 契约不足 → 补契约 **+ 补对应测试用例**。
  - 实现不足 → 补实现。

## 测试脚本（开发彻底结束后）
- 写 **api 测试脚本 + e2e 测试脚本**。
- **每个需求独立**，不和过往混在一起。
- 脚本实现与测试用例**一一对照**，也需要**单独审核**。

## 回归
- 彻底结束后，**全部执行统一回归**。
- **但不能一上来就跑带过往的全量**——先本需求，再统一回归。

## 不通过 → 回改
- 测试不通过 → 回过头改，直到通过。

## 解耦 + 可跳过（贯穿）
- 每一步互不耦合；用户可跳过任意一步（没测试用例、没脚本……）。
- 这与 `test-case` 现状"松耦合、不强制存在"一致，向整条链扩展。

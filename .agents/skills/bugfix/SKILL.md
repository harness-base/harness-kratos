# 修 bug（bugfix）

发现 bug 别直接改。本 skill 保证：根因定位、修得对、不回归、留下教训。

## 何时用 / 何时不用
- 用：bug / 测试失败 / 行为与预期不符 / 线上异常。
- 不用：新功能 / 用户可见行为变化（走 `feature-delivery`）；纯重构。

## 步骤
1. **复现**：找到稳定复现路径；**不能复现先别改**。
2. **定位根因（不猜）**：把症状写成「X 本该发生却没有」→ 沿调用 / 数据链路逐环查 → 定位断点。读代码 + 加日志/断点，找到**真因**不是表象。
3. **先写守护测试**：用**能复现 bug** 的测试锚住——改之前它必须**红**（mutation 自证：修复前红、修复后绿，才算抓对了点）。
4. **修**：改根因不改表象；最小改动。
5. **验证**：守护测试转绿 + 跑相关测试 / 验证（按 `workspace/verification.yaml` 路由）+ 确认没连带破坏别处。
6. **防回归 + 记 lesson**：守护测试入库；坑记 `tasks/lessons.md`（错在哪 / 怎么防 / 怎么更早发现），反复出现的晋升成规则（rule-0007）。

## 硬规则
- 不复现不开修；不猜根因（链路诊断定位）。
- 修复必须有**能复现 bug 的守护测试**（先红后绿、load-bearing），不许为通过牵强。
- `blocked`/`skipped` 不算修好（rule-0002）；没真验证不声称修复（rule-0003）。

## 演进（rule-0007）
bugfix 流程变化时回顾本 skill。

# 写代码（hc-dev）

写代码的**统一入口**——功能、工程代码、重构、改 bug 都走它。主线：**想清楚 → 列 plan → 你确认 → 写（全程不假设、决策点问你、防技术债）→ 挑刺（对抗 review，循环到无 bug）→ 提醒你测**。**纪律全程常开，仪式按级别伸缩**（默认我按任务轻重自判，你也可以指定级别）。借用 superpowers（brainstorming / writing-plans / TDD / review）与收尾 eval，**引用不重写**。依据见 ADR-0009。

## 何时用 / 何时不用
- 用：写 / 实现 / 改 / 重构代码；改 bug；做个功能。
- 不用：**产出**需求（走 `hc-prd`）；**产出**测试用例（走 `hc-test`）；纯文档 / 规则 / git 操作（各自 skill）。

## 全程常开纪律（两级都有，不可省）
- **不假设**：信息不足 → 去查事实源（仓内代码 / 资产 / 文档）或**摆出来问你确认**，绝不拿"已有信息 + 想要的结果"硬编（rule-0008）。
- **决策点停下问你**：方案分叉、取舍、不可逆动作前，**停下来问**，允许多轮沟通——不自作主张猜测。
- **防技术债**：边写边盯——别留 TODO 黑洞、别复制粘贴、别绕过既有抽象图省事；发现欠债当场记或当场还。
- **验证如实**：`blocked` / `skipped` 不当 pass（rule-0002）；没真实运行证据不声称完成（rule-0003）。
- **写完提醒你测**：自查与挑刺过后，明确告诉你"建议你跑/测哪些"，把人工测试交回给你。

## 两级（默认我按升级信号自判；你指定就按你的）

**升级信号**（命中任一 → 走深度，否则常规）：动核心逻辑 / 对外接口 / 不可逆操作（删数据、改 schema、迁移）/ 需要写决策记录（ADR）/ 关联多且不明确 / 需要读很多才能动手 / 用户可见的需求行为变化。

> **回指 hc-tech-design（边界指针）**：若本次涉及对外接口 / 数据模型的设计决策、且还没有 `docs/designs/<id>/` 契约 → 先走 `hc-tech-design` 产出技术设计 + 接口契约，再按契约实现（接口/schema 怎么定是设计阶段的事，hc-dev 按既定契约落码）。

### 常规级（小而明确、关联少、不用来回拉扯）
1. **列 plan** → 你点头才动手。
2. **写**（常开纪律全程在）。
3. **挑刺**：派 **1-2 个 `hc-code-reviewer` 子 agent** review，**循环到无 bug**（改→再 review→到干净）。
4. **提醒你测**。
- 不强制 brainstorm、不强制 eval。

### 深度级（动核心 / 接口 / 不可逆 / 要写 ADR / 复杂）
1. **brainstorm**：把"做成啥"聊清楚（用 `superpowers:brainstorming`，关键抉择问你）。
2. **正式 plan**：用 `superpowers:writing-plans` 出实现计划；**用户可见功能先立需求包**（`docs/features/`，rule-0001：未就绪 MUST STOP）。你确认才动手。
3. **写**（决策点必问；优先 TDD：先写能失败的测试再实现）。
4. **挑刺**：**多个不同视角 `hc-code-reviewer` 对抗 review**（correctness / 安全 / 技术债 / 边界…），每条独立证伪，**审→修→再审，循环到零**。
5. **收尾 eval**（rule-0005，用 hc-eval 子 agent，免 key）。
6. **提醒你测**。

## 改 bug 子模式（借自原 bugfix；两级通用）
1. **复现**：找稳定复现路径；**不能复现先别改**。
2. **定位真根因（不猜）**：把症状写成「X 本该发生却没有」→ 沿调用 / 数据链路逐环查 → 定位断点，改根因不改表象。
3. **先写能复现 bug 的守护测试**：改之前它必须**红**（mutation 自证：修复前红、修复后绿，才算抓对点；rule-0009）。
4. **修 → 验**（守护测试转绿 + 跑相关验证，确认没连带破坏）。
5. **防回归 + 记 lesson**：守护测试入库；坑记 `tasks/lessons.md`，反复出现的晋升成规则。

## 重构 / 迁移子模式（两级通用）
- **重构**：**行为保持**——先有覆盖既有行为的测试（特征 / golden 测试）再改；最小等价改写；挑刺重点放**等价性 + 回归**（行为有没有变）。
- **迁移**（不可逆，走深度级）：逐条对 `git show HEAD:<file>` / 原始来源**核源、禁凭记忆**（防"无损迁移"实为偷改，见 lessons 2026-06-26）；先确保可回滚，再迁，验证前后等价。

## 挑刺（对抗 review）怎么派
- **Claude Code**：用 workflow 编排——`agent(..., {agentType: 'hc-code-reviewer'})` 并行 / 循环（常规 1-2 个、深度多视角对抗到零）。
- **Codex**：用其原生子 agent 机制派同名 `hc-code-reviewer`（可能串行，角色一致）。
- **想要云端多 agent 深审**：提醒你跑 `/code-review ultra`（云端、计费、**只能你触发**，我启不了）。
- 复用共享子 agent `hc-code-reviewer`（`.claude/agents/` + `.codex/agents/` 双栈，免 key），不为单 skill 另造。

## 硬规则
- 写代码前先有 plan 且经你确认；**不假设、决策点必问**（贯穿，见上）。
- 用户可见功能改动：先立需求包并就绪才动业务代码（rule-0001，MUST STOP）。
- 修复 / 关键保证必须有**能复现 + load-bearing 的守护测试**（rule-0009），不许为通过牵强。
- 验证如实分类，无证据不声称完成（rule-0002 / 0003）。
- 大改（写了 ADR / 立了 feature）回顾相关 skill（rule-0007）；深度级收尾过 eval（rule-0005）。

## 演进（rule-0007）
开发流程 / 级别划分 / 挑刺机制变化时回顾本 skill（连同 `.claude/agents/hc-code-reviewer.md`、`.codex/agents/hc-code-reviewer.toml`）；改完同步 `version` / `last_reviewed`，跑 `bash scripts/skills-index.sh`。

# 需求产出（PRD + 交互原型）

`feature-delivery` 管"实现需求"，本 skill 管它的**上游——产出需求**：先把"做什么"想清楚（一份可观测验收的 PRD + 一套能点的原型），再由 feature-delivery 接力"怎么做"。**两者松耦合**：feature 实现体系不强依赖 PRD 存在，但有 PRD 时可平滑衔接（一份 PRD 派生 1..N 个 feature 包）。

## 何时用 / 何时不用
- 用：要从模糊想法或现有系统理出需求；要 PRD / 原型；产品方向、页面、流程得先想清楚再动手。
- 不用：需求已清晰、只是开发切片（直接走 `feature-delivery`）；纯 bug 修复 / 控制面改动。

## 步骤
1. **现状勘察（可选）**：若涉及现有系统/前端，先只读扫一遍页面与流程，作为原型一致性的事实源——轻量即可，不为"还原前端"单开重活。
2. **引导式多轮对话**：分轮澄清，别一次糊一堆问题——用户与要解决的事(JTBD) → 页面与流程 → 数据 → 四态(空/加载/错误/成功)+边界 → 验收目标 → 非目标。关键抉择用 `AskUserQuestion`，开放探索用对话。**不静默假设**：假设必须摆出来让用户确认。迭代到画面够清。
3. **产出 PRD**：用 `templates/prd.md`，落 `docs/prds/<id>/prd.md`，登记 `docs/prds/index.yaml`。每条验收目标**可观测可验证**（不是"做完了"）。
4. **产出交互原型**：可点的 HTML，落 `docs/prds/<id>/prototype/`——浏览器直开、mock 数据、无后端；原型即**视觉 + 交互规格一体**（设计稿和原型合一）。**若项目已有前端，原型与其内容/风格保持一致**。
5. **对抗式完整性评审**：PRD + 原型对标准自查（验收目标可观测吗？四态齐吗？流程闭合吗？原型能点通主流程吗？假设都确认了吗？）——需要时用多 agent 对抗证伪，迭代收敛。
6. **交接**：PRD 就绪后可派生 feature 包（`feature-delivery`）。衔接不强制，feature 体系可独立使用。

## 硬规则
- 引导式、多轮、收敛；**不静默假设**（摆出来确认）。
- 验收目标**可观测可验证**，不是"做完了"。
- 原型**能点通主流程**；项目有现有前端则与之内容/风格一致。
- 产物**独立存放**于 `docs/prds/`，与实现体系**松耦合**——不绑死 feature 体系。

## 门禁（rule-0010 / 考题 013）
- 产出 PRD 须满足 **rule-0010**（PRD 产出标准：验收可观测、范围闭合、四态齐、原型可点通、假设确认、可追溯、登记不漂移）。
- 结构（登记一致 + prd.md 必备章节）由 `scripts/prds-audit.sh` 机器校验，进 `make verify`。
- 收尾跑 eval **考题 013**（PRD/原型完整性与可观测性）。
- 但**不强制 PRD 必须存在**（与实现体系松耦合，ADR-0003）——门禁只在产出 PRD 时适用。

## 演进（rule-0007）
PRD / 原型的流程或标准变化时回顾本 skill。

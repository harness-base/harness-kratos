# PRD 写作指南

> `prd-elicitation` 第 3 步用。约束产出一份**人类可读、业务逻辑清晰、可追溯**的 PRD。按需读、不占常驻。
> 模板：`templates/prd.md`。上游事实视角：已确认的 `docs/prds/<id>/user-stories.md`。决策依据：ADR-0007。

## 一、底座
- 套 `templates/prd.md`，不另起结构。
- 必须**符合已确认的用户故事**（`user-stories.md`，approved 后才动 PRD）——PRD 不是重写需求，是把已对齐的需求展开成"做什么"。

## 二、功能点清单（必备）
- 列出本 PRD 所有功能点，每条 `FP-NN` + 一句话描述。`FP-NN` / `US-NN` 编号约定：两位补零、本批唯一且连号、一经引用不复用（见 `templates/user-story.md`）。
- **三级双向可追溯（软要求，目标覆盖 100%）**：
  - 每个用户故事 `US-NN` → 至少一个功能点 `FP-NN`（无遗漏）；
  - 每个功能点 `FP-NN` → 能追到某 `US-NN`（无凭空）；
  - 每个功能点 → PRD 正文有对应描述（不漏讲）。
- 在 PRD 里维护一张 `US ↔ FP ↔ 正文小节` 映射表自查；对抗评审（第 5 步）也查这条。
- **软**：暂不做机器闸，靠这张表 + 评审 + 本指南；复发/需要时再硬化（见 ADR-0007）。

## 三、人类可读 + 业务逻辑清晰
- 读者是人（产品 / 开发 / 评审）：用业务语言、少堆术语；先"为什么 / 做什么"，再细节。
- 业务逻辑成链：用户目标 → 流程 → 每步行为 / 规则 → 结果，能顺下来、不跳步。
- 每页四态（空 / 加载 / 错误 / 成功）+ 关键边界写清。

## 四、旧内容关联（接现有系统时）
- **找齐**：本需求触及的现有功能 / 页面 / 数据要找全，别漏关联。事实源 = **现有代码 / 现状**（不凭印象，见 ADR-0007 真相源优先级）。
- **描述旧场景**（不可能全讲清，按繁简分）：
  - 简单的 → 一句话带过；
  - 复杂的 → 附**引用来源**（文件 / 页面 / 文档路径），让用户自查，不在 PRD 里硬还原全部。
- 不臆造旧行为：拿不准就标"待核"或问用户。

## 五、产出后自查
- [ ] 每个 US 有 FP 覆盖、每个 FP 追到 US 且正文有描述、PRD 无孤儿内容（映射表齐）
- [ ] 业务逻辑顺得下来、不跳步
- [ ] 四态齐、关键边界写了
- [ ] 旧关联找齐、旧场景简 / 繁分别处理
- [ ] 假设都显式列出待确认（不静默假设）

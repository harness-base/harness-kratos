# 引导式把工程接进 harness（hc-onboard）

本 skill 管"**把一个工程按 harness 规范搭好、接进控制面**"——从零起的新项目挂进 `projects/`，配好最小骨架、决策记录、执行口接线，过审后交棒给开发链条（`hc-prd` → `hc-tech-design` → `hc-dev`）。依据 **ADR-0017**。

**形态 = 交互式引导**（主 agent 当**接入向导**），同 `hc-tech-design` 的「引导 → 用户确认 → 派 reviewer 对抗挑刺 → 回改 loop」——一条连贯的接入对话、逐步确认、不可拆并行。

**铁律（贯穿全程）**：选型 / 结构 / 规矩都是"**选择**"——**摆选项 + 讲取舍让用户拍板，不替用户定、不"看见啥落啥"**。接入向导的活是把路铺清楚、把取舍讲明白，拍板权归用户。

## ① 何时用 / 何时不用
- **用**：把**一个工程接进 harness**——从零起一个新项目（配项目名 / 栈 / 目标 → 搭最小骨架 → 记第一个决策 ADR → 接执行口 → 过审 → 登记），或（本期占位）把一个已有工程挂进控制面。用户说「接入项目 / 新建项目 / 挂个工程 / onboard / 起个新工程」时用。
- **不用**：**已接工程内**写功能 / 改 bug / 重构走 `hc-dev`；产**需求**走 `hc-prd`；产**研发方案 / 接口设计**走 `hc-tech-design`；产**测试用例**走 `hc-test`；固化一条**规则**走 `hc-add-rule`；改 **harness 本身 / 查 harness 漏洞**走 `hc-self-evolution`。
- 一句话边界：`hc-onboard` 只**把工程接进来、搭壳、接线**，**不搭代码结构、不写实现、不产需求 / 方案 / 用例**——那些是接进来之后的下游链条的活（见第 7 步交棒）。

## ② 入口分流（先问 新 / 老，本期只实现「新」）
接入向导第一句先分流，别急着往下走：

- **先问**：这是**从零起的新项目**，还是**已有代码的老工程要挂进来**？
- **新项目** → 走本 skill ③ 的 **7 步**（本期完整实现）。
- **老项目** → **本期占位**：明确告诉用户"老项目接入分支（存量代码盘点 / 现状 AGENTS 反向补 / 存量债标注等）**后续 ADR 补齐**，暂不走"，别拿新项目 7 步硬套老工程（老工程有存量代码 / 历史结构 / 隐性约定，流程不同）。指用户：眼下若只是要在老工程里改东西，直接走 `hc-dev` / `hc-tech-design`。

## ③ 新项目 7 步（每步：问什么 / 确认什么 / 落什么）
一条连贯的引导对话，主 agent 当接入向导，**每步先确认再落**。每步都守铁律（摆选项 + 讲取舍，用户拍板）。

### 第 1 步 · 收基本信息
- **问什么**：项目名（**kebab-case**，将来是 `projects/<名>/` 目录名）/ 目标（这工程要解决什么）/ 概述（一句话它是什么）/ 技术栈（框架 · 语言）。**怎么问由 agent 临场把握**——用户一次给全就不必逐条追；给得零碎 / 含糊就补问，别拿"已有信息 + 想要的结果"硬编（rule-0008）。
- **确认什么**：把收齐的四项**复述给用户确认**——尤其项目名是不是想要的 kebab、栈选型用户是否认可（栈有取舍就摆出来让用户拍，别默认替他选）。**项目名必须跟现有工程不撞**：`projects/` 是**多工程**目录（可能已挂别的工程），动手前先核 `projects/` 现有目录名 + `workspace/verification.yaml` 现有 `name`，撞了让用户另起一个唯一的 kebab 名，绝不覆盖 / 混进现有工程。
- **落什么**：**先不落文件**，把确认过的信息作为后续步骤的输入。

### 第 2 步 · 搭最小骨架（只搭壳、不搭代码结构）
- **问什么**：这一步主要是"搭"不是"问"，但**红线 / 指针要不要项目专属项**得跟用户核（如这项目有没有额外红线）。
- **确认什么**：给用户看将要建的骨架清单，确认了再写。
- **落什么**：
  - `projects/<名>/` 目录；
  - 精简 **`AGENTS.md`**——只放**项目红线 / 技术栈 / 指针**（指向控制面根 `AGENTS.md` 与产物区），**不复刻**全局规则、不堆流水账；
  - 同级 **`CLAUDE.md` shim**（`@AGENTS.md`，让 Claude Code 就近发现）。
- **红线（本步命根）**：**只搭壳、不搭代码结构**。目录布局 / 分层 / 包结构 / 脚手架**是 `hc-tech-design` / `hc-dev` 的活**，接入 skill **不越界**——建了 `projects/<名>/` + `AGENTS.md` + `CLAUDE.md` 就停，别顺手 `mkdir src/ internal/ cmd/`、别生成脚手架代码。

### 第 3 步 · 记第一个决策 ADR
- **问什么**：把第 1–2 步定下的**选型 / 结构为什么这么定**跟用户对齐——哪些是用户拍的、备选是什么、为什么排除。
- **确认什么**：ADR 的决策 / 备选 / 理由跟用户核对，别替用户编理由。
- **落什么**：一条**项目自己的决策记录 ADR**（落项目的决策记录处，具体路径由本步与用户约定 / 沿用项目约定），写清：**背景 → 决策 → 备选 + 为什么排除 → 影响**。这是这个项目"第一个决策留痕"，往后决策接着记。

### 第 4 步 · 接执行口（接线、不接假内容）
把项目的执行口**接上线**（不接假内容、不静默空）：

- **问什么**：这项目将来怎么验证——verify（项目自检）/ unit（单测）/ api（接口测）/ e2e（端到端）/ sandbox（沙箱）**这几条接入点，每条现在能接实吗**？将来脚本放哪、叫什么命令？
- **确认什么**：每条接入点的**三态取值**（见 ④）跟用户逐条确认——现在有真命令就填真命令；接不了就填 `PENDING:` 并写清楚为什么现在空 / 补的条件；这项目压根不需要就填 `N/A:` 并写理由。
- **落什么**：
  - 在 `workspace/verification.yaml` 给项目**占一条**（verify / unit / api / e2e / sandbox 各按三态取值）——**注意隔离纪律**：本 skill **只教用户怎么标、教 reviewer 怎么审**，实际写 `workspace/verification.yaml`、接 CI、建机检脚本是**主 agent 串行接线**做（本 skill 不代跑 index/regen、不代改 workspace / CI）；
  - 约定 **CI 认得它**（口子留好）；
  - 约定**将来脚本放哪、叫什么命令**（命令名先约定，**脚本本体先占位**）。
- **sandbox 特别说明**：sandbox 复杂，**下一轮单独建 `create-sandbox` skill**；本 skill 里 sandbox **只留占位**（三态里多半是 `PENDING:` 或 `N/A:`），不在这里搭沙箱。**verify / ci 是本 skill 内部小步骤**、不单独成 skill。

### 第 5 步 · 对抗评审搭出来的骨架
- **派 `hc-onboard-reviewer`** 子 agent 对抗挑刺搭出来的骨架（骨架完整性 / 三态占位是否合规 / 隔离边界有没有踩 / 是否越界搭了代码结构），**回改到过**（详见 ⑤）。

### 第 6 步 · 收尾
- **`make verify` 绿**：控制面自检通过（含结构 / 文档 / 索引）。
- **项目登记好**：项目在控制面被认得（登记 / 索引由主 agent 串行跑，本 skill 不代跑）。
- **确认什么**：把"verify 绿 + 登记好"的**真实证据**摆给用户，没真跑通不许声称通过（rule-0002 / rule-0003）。

### 第 7 步 · 交棒
- 骨架**过审 + verify 绿**后，**指路下游**：往后开发走 `hc-prd`（产需求）→ `hc-tech-design`（研发方案 / 接口设计）→ `hc-dev`（写代码）。
- **第一个需求包按 rule-0001 立**：接入完成不等于可以直接写业务码——**用户可见的需求 / 行为 / 验收目标**要先在 `docs/features/` 立需求包并就绪，未就绪 MUST STOP。交棒时把这条讲给用户。

## ④ 占位三态规矩（关键：占位不许静默空）
第 4 步每个接入点（verify / unit / api / e2e / sandbox）的值**必是三态之一**——**目标：占位看得见 + 绕不过去，防"开发过了没补"**（同 `test-cases-audit` 的「无 · 理由」逃生口思路）：

1. **真命令**（已接实）——直接写将来能跑的命令。
2. **`PENDING: <为啥现在空 / 补的条件>`**（待接实）——例：`PENDING: 首个接口落地后补 api 冒烟`。**`make verify` 会 warn 提醒**（机检脚本主 agent 另建），同时**项目 `AGENTS.md` 里留一条"待补"记录**，让 agent 以后进来看得见、绕不过去。
3. **`N/A: <理由>`**（这项目不需要这项）——例：`N/A: 纯 CLI 工具、无对外接口`。

**静默空 / 留白 / 裸 `TODO` = 红**。skill 只**教用户这么标、教 reviewer 这么审**；机检脚本本体由主 agent 另建。

## ⑤ 对抗评审（派 hc-onboard-reviewer 到过）
骨架搭完（第 5 步），**派 `hc-onboard-reviewer` 子 agent 对抗挑刺**，**回改到过**（同 `hc-tech-design` 单 reviewer 线性 loop）：
- reviewer **只评不改**（无 Write），出结构化清单（位置 / 严重度 / 问题 / 证据 / 怎么补）；
- 主 agent（接入向导）据清单**回改 → 复审 → 到零缺陷**。
- **怎么派**：**Claude Code** 用 workflow / Task（`agent(..., {agentType:'hc-onboard-reviewer'})` 循环，会话模型）；**Codex** 派**同名**双栈 reviewer。
- **reviewer 审什么**（总览，约束本体在 reviewer 子 agent 上下文）：① **骨架完整**（`projects/<名>/` + 精简 AGENTS.md + CLAUDE.md shim 齐、AGENTS.md 只有红线 / 栈 / 指针不复刻全局）② **没越界**（只搭壳、**没搭代码结构 / 脚手架**）③ **三态占位合规**（verify / unit / api / e2e / sandbox 每条是真命令 / `PENDING:` / `N/A:` 之一，**无静默空 / 裸 TODO**，PENDING 有条件、N/A 有理由，AGENTS.md 有"待补"记录）④ **第一个 ADR 有据**（决策 / 备选 / 理由齐，别替用户编）⑤ **隔离没搞反**（骨架里写满真实领域词是**对**的；反过来 skill / reviewer / 模板不掺项目内容，见 ⑥）。

## ⑥ 硬规则
- **只搭壳、不越界**：建 `projects/<名>/` + 精简 AGENTS.md + CLAUDE.md shim 就停，**不搭代码结构 / 分层 / 脚手架 / 实现**（那是 `hc-tech-design` / `hc-dev` 的活）。
- **多工程隔离**：`projects/` 可能已有别的工程——接新工程**只动自己那份**（建自己的 `projects/<名>/`、给 `verification.yaml` **append** 自己那条、只落自己项目的 ADR / 规则），**绝不碰、不覆盖别的工程**的目录 / `AGENTS.md` / verification 条目 / 决策记录；项目名与路径**不与现有工程撞**（第 1 步先核）。
- **全程用户确认、不替用户定**：选型 / 结构 / 规矩 / 接入点取值都**摆选项 + 讲取舍让用户拍**，不"看见啥落啥"、不静默替用户拍板；不确定就查 + 问（rule-0008）。
- **占位不静默**：接入点守三态（真命令 / `PENDING: <条件>` / `N/A: <理由>`），**静默空 / 留白 / 裸 TODO = 红**；PENDING 同步在项目 AGENTS.md 留"待补"记录。
- **通用 / 项目隔离别搞反**（rule-0015）：**skill 本体 / hc-onboard-reviewer / 模板**必须**通用、不掺具体项目内容**（中性占位，不预设某语言 / 框架 / 多租户 `tenant_id`）；但本 skill **产出的项目骨架**（`projects/<名>/AGENTS.md`、第一个 ADR、`verification.yaml` 项目条目）**本就项目专属、写满真实领域词是对的**。方向别反：通用的归资产、具体的归产物。
- **接线归主 agent**：本 skill **不代改** `workspace/verification.yaml` / CI、**不代建**机检脚本、**不代跑** index / regen / ADR 定稿——skill 教用户怎么标、教 reviewer 怎么审，接线由主 agent 串行做。
- **收尾不假完成**：`make verify` 绿 + 登记好要有**真实证据**（rule-0002 / rule-0003）；交棒时提醒第一个需求包按 rule-0001 立。
- **对抗评审到过**：派 `hc-onboard-reviewer` 回改到零缺陷。

## ⑦ 演进（rule-0007）
接入流程 / 骨架结构 / 三态占位规矩 / 评审机制 / 分流（新增老项目分支）变化时回顾本 skill（连同 `.claude/agents/hc-onboard-reviewer.md`、`.codex/agents/hc-onboard-reviewer.toml`、及将来的接入模板）；改完同步 `version` / `last_reviewed`，跑 `bash scripts/skills-index.sh`（本步由主 agent 串行做，skill 本身不代跑）。

---

> 本 skill 为通用引导，项目大小可按需伸缩（小工程接入几步从简、大工程逐步做实）；但**门槛**（先分流 / 全程用户确认 / 只搭壳不越界 / 占位三态不静默 / 对抗评审到过 / 收尾真实证据）不省。
